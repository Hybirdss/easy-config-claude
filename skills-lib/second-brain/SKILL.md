## 모드 1: SOP / Runbook 작성

**철학:** 아무도 50페이지 문서는 안 읽는다. 스캔 가능하고, 실행 가능하고, 오해할 수 없게.

### 정보 수집 (AskUserQuestion)

```
1. 이 문서의 목적은? (장애 대응 / 배포 절차 / 온보딩 / 기타)
2. 독자는 누구? (주니어 개발자 / 시니어 / 비개발자)
3. 핵심 단계를 알고 있어요? 아니면 같이 정리할까요?
```

### 문서 구조

```markdown
# [문서 제목]

> **마지막 업데이트:** [날짜] | **담당자:** [이름] | **검토 주기:** [분기/반기]

## 언제 이 문서를 쓰나요
[한 줄로 — 상황/트리거 조건]

## 전제 조건
- [ ] [접근 권한 A]
- [ ] [도구 B 설치]
- [ ] [알아야 할 것 C]

## 단계

### Step 1: [동사로 시작하는 명확한 제목]
**목표:** [이 단계가 끝나면 뭐가 됐어야 하는지]

```bash
# 실행 명령 (복붙 가능하게)
command --flag value
```

**확인:** [어떻게 잘 됐는지 확인하나]
**실패 시:** [이 단계 실패하면 어떻게]

[Step 2, 3...]

## 완료 기준
- [ ] [체크리스트 항목 1]
- [ ] [체크리스트 항목 2]

## 롤백 (해당 시)
[되돌려야 할 때 어떻게]

## 자주 발생하는 문제
| 증상 | 원인 | 해결 |
|------|------|------|
```

### 작성 원칙
- 모든 단계는 동사로 시작 (확인하다 X → 확인한다 O, 또는 "Run", "Check", "Update")
- 숫자와 이름을 구체적으로 ("적당히" X → "5분 이내" O)
- 성공 기준 명시 ("완료" X → "HTTP 200 응답 확인" O)
- 담당자·검토 주기 필수 (문서 부패 방지)

## 모드 2: 브랜드 보이스 문서화

브랜드/제품/팀의 커뮤니케이션 톤을 정의하는 문서 작성.

### 정보 수집

```
1. 이 브랜드/팀을 한 문장으로?
2. 경쟁사나 참고 브랜드가 있어요? (우리랑 다른 점이 뭔가요?)
3. 절대 쓰면 안 되는 단어나 표현이 있어요?
4. 대상 독자의 나이/직업/관심사?
```

### 문서 구조

```markdown
# [브랜드명] 보이스 가이드

## 우리 브랜드는 ___이다
[형용사 3개로 정의]

## 말하는 방식
| 이렇게 | 이렇게는 X |
|--------|-----------|
| [구체적 예시] | [피해야 할 표현] |

## 금지 어휘
[절대 쓰지 않는 단어 목록과 이유]

## 상황별 예시
### 에러 메시지
### 온보딩 문구
### 마케팅 카피
```

## 모드 3: 지식 캡처 (TIL)

오늘 배운 것을 `~/.claude/memory/knowledge.md`에 기록.

```bash
KNOWLEDGE_FILE="$HOME/.claude/memory/knowledge.md"
mkdir -p "$(dirname "$KNOWLEDGE_FILE")"
[ ! -f "$KNOWLEDGE_FILE" ] && echo "# Knowledge Base\n" > "$KNOWLEDGE_FILE"
```

포맷:
```markdown
## [날짜] — [주제]
**컨텍스트:** [왜 이걸 배웠나]
**핵심:** [한두 문장 요약]
**참고:** [링크/파일]
```

Claude가 다음에 관련 주제가 나오면 이 파일을 Read해서 이전 학습을 참조한다.
